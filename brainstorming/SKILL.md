# Brainstorming Skill

在任何创造性工作（创建功能、构建组件、添加功能或修改行为）**之前**必须使用。在实施之前探索用户意图、需求和设计。

## Core Checklist (9 Steps)

每一步包含触发条件、操作、成功标准和失败分支：

### Step 1: 探索项目上下文
- **操作**：检查项目根目录文件列表、`docs/` 目录、最近的 git log (最近 5 条)
- **成功**：了解当前技术栈、已有模块、代码风格
- **失败**：如果项目目录为空 → 跳至 Step 3 直接澄清需求

### Step 2: 🔴 提供可视化伴侣
- **触发条件**：话题涉及 UI/布局/架构图/视觉对比
- **操作**：单独发送可视化伴侣提议，不与澄清问题合并。等待用户回应
- **失败**：如果用户拒绝 → 用文字描述替代，继续 Step 3

### Step 3: 提出澄清问题
- **操作**：一次一问，每轮 2-4 个选项或开放式。覆盖：目的、约束、成功标准
- **成功**：用户意图清晰，可以进入方案设计
- **失败**：如果 3 轮后意图仍不清晰 → 主动总结已知信息，基于假设提出 provisional 方案并标注假设风险

### Step 4: 提出 2-3 种方案
- **操作**：每种方案包含：核心思路、优势、劣势、适用场景。先推荐最优方案
- **成功**：用户选定或倾向于某一方案
- **失败**：如果用户都不满意 → 追问不满意的具体原因，重新生成；最多 2 轮

### Step 5: 呈现设计方案
- **操作**：按复杂度分节（架构→组件→数据流→错误处理→测试），每节后问「这部分对吗？」
- **成功**：每节获得用户认可
- **失败**：如果任一节被否定 → 修改该节，不重头开始；如果 2 节以上被否定 → 回到 Step 4 重新考虑方案

### Step 6: 编写设计文档
- **操作**：写入 `docs/designs/YYYY-MM-DD-<topic>-design.md`，git add + commit
- **成功**：文件写入成功且已提交
- **失败**：如果 `docs/designs/` 目录不存在 → 先 `mkdir -p docs/designs/` 再写入

### Step 7: 规格自查
- **操作**：逐项扫描：TBD/TODO 占位符、内部矛盾、范围边界、歧义双解
- **成功**：0 问题发现 → 进入 Step 8
- **失败**：发现问题 → 修复后重扫，最多 2 轮；仍有问题 → 标注已知风险后进入 Step 8

### Step 8: 🔴 用户审核
- **操作**：> "设计规格已写入 `<path>`。请审核，需要任何更改吗？"
- **成功**：用户明确批准
- **失败**：用户要求修改 → 执行修改 → 回到 Step 7 重扫 → 再次审核。最多 3 轮

### Step 9: 过渡到实施
- **操作**：确认设计文档已提交，结束 brainstorm 流程，开始实施
- **失败**：如果用户在此阶段提出新需求 → 回到 Step 3 重新澄清，新需求纳入设计

## Key Rules

### 🔴 HARD-GATE · 🛑 STOP
> 在呈现设计方案并获得用户批准之前，不要编写任何代码、搭建任何项目或采取任何实施行动。这适用于每个项目，无论看起来多简单。
>
> **触发时机**：用户说了任何带「写/做/实现/搭建/创建」的请求
> **动作**：拒绝执行，进入 brainstorm 流程。设计确认后才放行。

### 🔴 CHECKPOINT：关键决策点

以下节点必须暂停等用户确认，不可自行推进：

| # | 检查点 | 位置 | 确认方式 |
|---|--------|------|----------|
| 1 | 澄清完成，进入方案 | Step 3→4 | 总结理解，问「我理解正确吗？」|
| 2 | 方案选定，进入设计 | Step 4→5 | 用户明确选定某一方案 |
| 3 | 每节设计确认 | Step 5 内部 | 「这部分对吗？」|
| 4 | 设计文档审核 | Step 8 | 用户明确批准规格文件 |
| 5 | 进入实施 | Step 9 | 用户确认可以开始实施 |

## ⛔ 反模式与禁止操作

以下行为在 brainstorm 阶段**绝对禁止**：

| # | 禁止操作 | 原因 | 正确做法 |
|---|----------|------|----------|
| 1 | 未获批准就写代码/搭项目 | 跳过设计直接实施 = 浪费 | 必须过完 9 步，HARD-GATE 不破 |
| 2 | "太简单不需要设计" | 简单项目正是未经审视的假设导致最多浪费的地方 | 设计可以简短（几句话），但必须呈现并获批准 |
| 3 | 一次问多个澄清问题 | 用户会被淹没，答案质量差 | 一次一问，循环收敛 |
| 4 | 只给一种方案 | 剥夺用户的选择权 | 至少 2 种方案 + 推荐 |
| 5 | 跳过用户审核直接实施 | 设计是给用户看的，不是自嗨 | Step 8 必须等用户明确批准 |
| 6 | 设计未经 git 提交就实施 | 失去追溯链 | Step 6 必须 commit |
| 7 | 在澄清不足时强行推进方案 | 基于错误假设的设计是废纸 | Step 3 最多 3 轮仍不清晰 → 基于假设出 provisional 方案并标注风险 |
| 8 | 对否定反馈整节重写 | 浪费时间，丢已确认内容 | 只改被否定的部分 |
| 9 | 设计文档堆砌实现细节 | 设计≠代码，设计是"做什么"不是"怎么做" | 聚焦架构/数据流/接口/错误策略，不写代码片段 |

## Design Principles

### Understand the Idea
- **上下文检查**：列出项目根目录文件 + `docs/` 结构 + `git log --oneline -5`
- **范围评估**：如果请求涉及 3+ 独立子系统，先帮用户拆成子项目再逐个处理
- **澄清方式**：一次一问，优先给 2-4 个具体选项。开放式问题仅在无可选项时使用
- **收敛条件**：目的、约束、成功标准三项都明确 → 进入方案阶段

### Explore Approaches
- **方案数量**：必须提 2-3 种方案，不允许只给一种
- **方案格式**：每种方案 = 核心思路（1句）+ 优势（2-3条）+ 劣势（1-2条）+ 适用场景
- **推荐**：先展示推荐方案，说明理由。然后列出备选

### Present Design
- **分节结构**：架构→组件→数据流→错误处理→测试。简单项目每节 2-3 句话，复杂项目每节 ≤300 字
- **验证节奏**：每节结尾加「这部分对吗？」等待确认
- **不写代码**：设计文档描述结构和逻辑，不包含代码片段

### Isolation & Clarity
- 将系统分解为独立单元，每个单元需回答：做什么、如何使用、依赖什么
- 测试标准：不读内部实现能否理解功能？改内部实现是否会破坏消费者？任一条不满足 → 重新设计边界

### Working in Existing Codebases
- 改前先读：找到最近修改过的相关文件，理解现有模式
- 如果现有代码有影响工作的 bug → 纳入设计作为前置修复项
- 禁止提出与当前目标无关的重构

## After Design

### Documentation
1. 确保 `docs/designs/` 目录存在（不存在则 `mkdir -p`）
2. 写入 `docs/designs/YYYY-MM-DD-<topic>-design.md`
3. `git add` + `git commit -m "docs: add design for <topic>"`

### Spec Self-Check（内联执行，不派子代理）
逐项扫描并自动修复：

| 检查项 | 扫描内容 | 发现问题时 |
|--------|----------|------------|
| 占位符 | `TBD`、`TODO`、`???`、留空的表格单元格 | 填入具体值 |
| 一致性 | 架构描述 vs 组件列表是否匹配 | 以架构描述为准，修正组件列表 |
| 范围 | 是否涉及 3+ 独立子系统 | 拆分为多个子设计文档 |
| 歧义 | 同一需求是否可被两种方式解释 | 选择一种并明确标注决策 |

扫完输出：`✅ 自查通过，0 问题` 或 `⚠ 发现 N 个问题，已修复`

### 🔴 User Review Gate
> "设计规格已写入 `<path>`，自查通过。请审核，需要任何更改吗？"

等待用户明确回复「可以」「没问题」「批准」等同义确认词。仅在批准后才过渡到 Step 9。

## Visual Companion

基于浏览器的可视化工具，详见 `visual-companion.md`。用于展示模型、图表、选项对比。

### 启动（Windows）
```bash
cd ~/.workbuddy/skills/brainstorming
bash scripts/start-server.sh --project-dir <当前项目路径> --foreground
```
输出 JSON 含 `url`、`screen_dir`、`state_dir`。告诉用户打开 URL。

### 使用循环
1. **写 HTML 片段**到 `screen_dir`（只用 Write 工具，语义化文件名如 `layout.html`，不复用文件名）
2. **告诉用户**看浏览器 + 做选择
3. **读 `state_dir/events`** 获取用户点击数据
4. **迭代或推进**

### 关闭
```bash
bash scripts/stop-server.sh $SESSION_DIR
```

### 何时用
- 浏览器：UI 布局/架构图/视觉对比/状态机
- 终端：需求问题/概念选择/权衡列表/技术决策
